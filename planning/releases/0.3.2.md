# faststream-redis-timers 0.3.2 — internal hardening, no behavior change

A maintenance release: two significant internal refactors that make the timer
protocol and the poll loop easier to test and reason about, plus packaging and
docs/CI improvements. **No public-API or observable-behavior change** — upgrading
is a drop-in.

## Internal refactors

- **TimerStore.** The Redis timer protocol (timeline sorted-set + payloads hash +
  lease Lua) now lives behind one module, `faststream_redis_timers/store.py`.
  Five previously-scattered Redis touch-points (producer, subscriber poll/claim,
  message commit, broker inspection) and the topic-key derivation collapse behind
  a single interface; `subscriber/lua.py` is folded in and removed. Behavior is
  preserved, with one internal improvement: the cancel/commit paths now use a
  single atomic Lua round-trip instead of a `MULTI`/`EXEC` pipeline.
- **PollSchedule.** The subscriber's poll-loop backoff (idle/error ramps, jitter,
  caps) is extracted into a pure, table-tested `PollSchedule`
  (`faststream_redis_timers/subscriber/schedule.py`); `_consume` now holds only
  I/O and control flow. Backoff timing, caps, and reset rules are unchanged.

## Packaging

- Enriched PyPI metadata — keywords, classifiers, and project URLs.

- Documentation now published at
  <https://faststream-redis-timers.modern-python.org> via GitHub Actions Pages.

## Downstream

No action needed — there is no API or behavior change; pin floors are unaffected.
